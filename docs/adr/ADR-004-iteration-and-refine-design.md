# ADR-004: Iteration and Refine Design

**Status**: Accepted  
**Date**: 2025-02-10  
**Context**: Sprint 4 — Documentation & Mental Model

## Summary

This ADR records the design decisions behind PromptAura’s **task-based iteration loop**: why the product is task-based, why there is no manual prompt mode, and why **Gemini is the single prompt creator**. It complements ADR-002 (Gemini research pipeline) and ADR-003 (iteration-only playground) by making the rationale explicit for long-term maintainers and stakeholders.

---

## 1. Why task-based?

**Decision:** The user supplies a **task goal** (intent), not a raw prompt. The system is responsible for turning that goal into concrete prompts and parameters.

### Rationale

- **Intent vs implementation:** Most users care about “what I want” (e.g. “a cat in a hat, watercolor”) rather than the exact token sequence or model-specific parameter names. Task-based input matches that mental model and keeps the UI simple.
- **ModelSpec alignment:** Prompts are generated by Gemini using the same ModelSpec (guidelines, inputs/outputs, recommended ranges) that was produced during model research. Task-based flow ensures every prompt is spec-aware; free-form prompt input would bypass that and risk off-spec or poorly parameterized prompts.
- **Refine loop:** Refine only works if the system “owns” prompt authoring. The system needs to combine task + selected prompts + notes + output summaries into improved prompts. If users could paste arbitrary prompts, the refine step would be ill-defined and the value of “evolved” prompts would drop.
- **Consistency:** One input shape (task) for both generate and refine keeps the product easy to explain and to implement.

### Alternatives considered

- **Raw prompt input (manual mode):** Rejected; see ADR-003 and section 2 below.
- **Hybrid (task + optional prompt override):** Rejected; adds complexity and undermines the “single loop” story and refine semantics.

---

## 2. Why no manual mode?

**Decision:** There is no “manual mode” where the user types a single prompt and runs it once. The Playground is iteration-only (ADR-003).

### Rationale

- **Product focus:** The product is “discover the best prompt by iterating,” not “run one prompt.” Manual mode would dilute that and force the UI to support two different mental models and two code paths.
- **Clarity:** “One flow only” makes onboarding and documentation simple: everyone does task → generate → select → refine. No “when do I use manual vs iteration?”
- **Refine semantics:** Refine is defined over “previous iteration’s candidates + your feedback.” Manual runs don’t fit that model; they would require special cases (e.g. “refine from manual prompt”) and weaken the guarantee that refined prompts are built on prior system-generated candidates and run results.
- **Maintenance and testing:** One flow means one set of APIs, one UI path, and one story for tests and observability (e.g. iteration/runs in Admin).

### Consequences

- Users cannot “quickly test one prompt” without going through generate (or a previous refine). That is an accepted tradeoff for a single, clear iteration loop and better long-term outcomes via refine.

---

## 3. Why Gemini as the single prompt creator?

**Decision:** All candidate prompts (initial generate and every refine) are produced by **Gemini**. The execution provider (e.g. fal.ai) only runs the prompts returned by the system; it does not author them.

### Rationale

- **Single reasoning engine:** The same system that researches models and produces ModelSpec also writes prompts. It consistently uses the same guidelines, parameter ranges, and input/output structure, so prompts stay on-spec and comparable across iterations.
- **Task → prompts:** The user gives a task goal; Gemini turns it into N concrete prompts (and on refine, into M improved prompts using selections, notes, and output summaries). Users don’t need to know prompt engineering or model-specific parameters.
- **Refine quality:** Refine sends Gemini (i) selected prompts, (ii) user notes, and (iii) short summaries of run outputs (no raw JSON). That context is why refined prompts “evolve” in a meaningful way rather than feeling random (see Blok C / iteration context).
- **Separation of concerns:** Execution providers (fal.ai, etc.) are optimized for running jobs (image/video generation, queues, scaling). They are not built for structured, spec-aware prompt authoring. Gemini handles authoring; the execution provider is a runner only.
- **Contract and quality:** Prompts are generated from a single template/contract (e.g. `promptTemplates.ts`, Contract v2 with reasoning/tags), which makes behavior predictable and allows quality improvements in one place.

### Alternatives considered

- **Target model generates its own prompts (“self”):** Rejected for current scope. fal.ai image/video models don’t author prompts; keeping “Gemini only” avoids branching logic and keeps one mental model (“Gemini writes, provider runs”). If a future model natively supports high-quality, spec-aware prompt generation, a follow-up ADR can revisit.
- **fal.ai LLM for prompt generation:** Rejected; fal.ai’s strength is execution, not reasoning. Gemini is already used for research and has the ModelSpec context; using it for prompts keeps a single reasoning stack and better refine quality.

---

## Consequences (overall)

### Positive

- **Clear mental model:** Task → generate → select → refine, with Gemini as the only prompt author and the execution provider as the only runner.
- **Easier onboarding:** README and docs can describe one flow; “how it works” and “why Gemini” are answerable in a few paragraphs.
- **Maintainability:** One loop, one prompt source, one contract; fewer special cases and clearer ownership (e.g. observability: Gemini request id vs fal.ai request id).

### Negative / mitigations

- **No quick single-prompt run:** Mitigated by fast iteration (queue + polling) and by treating “generate 20” as the entry point.
- **Dependency on Gemini for prompts:** Mitigated by contract/templates and tests; failure modes and timeouts are documented and observable (Blok E).
